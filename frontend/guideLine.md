# 프론트엔드 기능 명세 가이드라인 (guideline.md)

본 문서는 기존 `FungameFE` 프로젝트의 코드를 분석하여 추출한 **순수 기능 및 비즈니스 로직 명세서**입니다. UI/UX 디자인 요소를 배제하고 데이터 흐름과 상태 관리에 집중합니다.

---

## 1. 핵심 기술 및 상태 관리

- **라우팅**: `react-router-dom` (상태 기반 리다이렉트 수행)
- **통신**:
    - REST API: `axios` (BaseURL: `VITE_API_BASE_URL`)
    - WebSocket: `@stomp/stompjs` + `sockjs-client`
- **전역 상태 관리 모델**: `useGameLogic` 커스텀 훅을 통한 중앙 집중식 상태 관리 (Status 기반 State Machine).
- **영속성(Persistence)**: `localStorage`를 사용하여 브라우저 새로고침 시에도 상태 복구.
    - `ums_nickname`: 사용자 닉네임
    - `ums_roomId`: 현재 속한 방 ID
    - `ums_status`: 현재 앱 상태 (`LOBBY`, `ROOM_LIST`, `WAITING`, `PLAYING`, `RESULT`)

---

## 2. 애플리케이션 상태(Status) 흐름 및 라우팅

앱은 `status` 값에 따라 5개의 주요 화면으로 강제 라우팅됩니다.

1. **`LOBBY` (경로: `/`)** -> 닉네임 설정
2. **`ROOM_LIST` (경로: `/rooms`)** -> 방 목록 및 생성
3. **`WAITING` (경로: `/waiting`)** -> 게임 대기실
4. **`PLAYING` (경로: `/game`)** -> 실제 게임 진행
5. **`RESULT` (경로: `/result`)** -> 최종 결과 화면

*※ 부트스트랩(초기화) 단계에서 `status`와 `roomId`를 확인하여 이전 세션으로 자동 복귀(Re-entry)하는 기능이 구현되어 있습니다.*

---

## 3. 화면별 세부 기능 명세

### A. 닉네임 페이지 (LOBBY)
- **기능**: 사용자 닉네임 입력.
- **동작**: 입력받은 닉네임을 로컬 스토리지에 저장하고 `ROOM_LIST` 상태로 전환.

### B. 방 목록 페이지 (ROOM_LIST)
- **기능**: 서버에 생성된 게임 방 목록 조회, 새 방 생성, 기존 방 입장, 닉네임 변경.
- **API 연동**:
    - **조회**: `GET /game/rooms`
    - **생성**: `POST /game/rooms` (방 제목, 최대 인원, 카테고리, 총 곡 수 전송). 성공 시 해당 방의 방장이 되어 `WAITING`으로 이동.
    - **입장**: `POST /game/rooms/{roomId}/join` (헤더에 `playerName` 인코딩하여 전송). 성공 시 `WAITING`으로 이동.
- **예외 처리**: 다른 방에서 이미 게임 중인 유저가 입장 시도 시 HTTP 409 Conflict 발생. 이 경우 서버가 반환한 `redirectRoomId`를 통해 강제로 진행 중인 방(`PLAYING`)으로 이동시킴.

### C. 게임 대기실 (WAITING)
- **기능**: 방에 입장한 플레이어 목록 동기화, 웹소켓 연결, 채팅, 방장 권한 처리(게임 시작).
- **API/WS 연동**:
    - **유저 목록 조회**: `GET /game/rooms/{roomId}/users`
    - **웹소켓 연결**: `/ws-quiz`로 STOMP 연결 및 `/subscribe/room/{roomId}` 구독.
    - **퇴장**: `POST /game/rooms/{roomId}/leave` 호출 후 `ROOM_LIST`로 상태 변경 및 구독 해제.
    - **게임 시작**: 방장일 경우에만 `POST /game/rooms/{roomId}/start` 호출 가능.

### D. 인게임 (PLAYING)
- **기능**: 타이머 동기화, 퀴즈 내용 표시, 유튜브 영상/음원 재생, 정답 입력, 실시간 순위 변동.
- **데이터 흐름**:
    - 정답 시도 및 채팅은 웹소켓 `publish` (`/publish/room/{roomId}/chat`) 로 전송.
    - `ROUND_START` 웹소켓 이벤트를 통해 전달받은 `videoURL`을 `react-youtube`를 이용해 재생.
- **API 연동**:
    - **실시간 랭킹 강제 동기화**: `ROUND_END` 이벤트 수신 시 `GET /game/rooms/{roomId}/play/rank`를 호출하여 점수 재정렬.
- **네비게이션 제한**: 게임 중 뒤로가기(`popstate`) 시도 시 경고 없이 강제 퇴장(`leaveRoom`) 처리됨.

### E. 결과 화면 (RESULT)
- **기능**: 최종 게임 결과 및 순위 출력.
- **동작**: 웹소켓 `GAME_RESULT` 이벤트를 통해 전달받은 순위 데이터를 파싱하여 출력. '로비로 돌아가기'를 통해 방에서 퇴장하고 `ROOM_LIST`로 복귀.

## 4. WebSocket 이벤트(Event) 처리 명세

서버로부터 수신하는 STOMP 메시지(`GameEvent`)에 따른 프론트엔드 상태 변경 규칙입니다.

| 이벤트 타입 | 주요 기능 / 상태 변경 |
| :--- | :--- |
| `PLAYER_JOIN` / `LEAVE` | 시스템 로그 출력 및 방 유저 목록(`fetchRoomUsers`) 재요회 |
| `HOST_CHANGE` | 새로운 방장 닉네임을 감지하여 본인의 `isHost` 권한 업데이트 |
| `CHAT` | 일반 채팅 메시지 로그 배열에 추가 |
| `GAME_START` | 게임 시작 임박 안내 메시지 출력 (실제 화면 전환은 안함) |
| `ROUND_START` | **`status`를 `PLAYING`으로 변경**, 유튜브 비디오 ID 교체, 라운드 정보 갱신 |
| `TIMER_TICK` | 남은 시간(UI 타이머) 동기화 |
| `CORRECT_ANSWER` | 특정 플레이어의 점수 실시간 갱신 |
| `ROUND_END` | 정답 및 승자 표출, 서버 랭킹 동기화 API 즉시 호출 |
| `GAME_RESULT` | **`status`를 `RESULT`로 변경**, 문자열로 전달된 최종 랭킹 파싱 및 저장 |

## 5. 기능 개선 및 사용자 경험 고도화 제안

기존 코드 분석을 통해 도출된, 실제 게임 플레이 품질을 높이기 위한 기능적 개선 사항입니다.

### A. 대기실 '준비(Ready)' 시스템 도입
- **현상**: 현재는 방장이 모든 인원의 상태와 무관하게 즉시 게임을 시작할 수 있음.
- **개선**: 
    - 일반 플레이어에게 '준비' 상태값 추가.
    - 모든 플레이어가 '준비' 완료 상태일 때만 방장에게 '시작' 버튼 활성화.
    - 방장이 시작을 눌렀을 때 인원 변동이 생기면 시작 시퀀스를 취소하는 예외 처리 로직 강화.

### B. 유튜브 재생 시점 강제 동기화 (Sync)
- **현상**: 네트워크 지연이나 중도 입장 시 플레이어마다 영상 재생 시점이 달라질 수 있음.
- **개선**: 
    - `TIMER_TICK` 이벤트 수신 시, 서버의 남은 시간과 현재 유튜브 플레이어의 재생 시간을 대조.
    - 오차 범위(예: 2초 이상) 발생 시 `player.seekTo()`를 통해 강제로 서버 시간에 맞춤.

### C. 정답자 발생 시 시각적/청각적 피드백 로직
- **현상**: 채팅 로그로만 정답을 확인할 수 있어 게임의 박진감이 떨어짐.
- **개선**:
    - `CORRECT_ANSWER` 이벤트 수신 시, 화면 중앙에 정답자와 메시지를 강조 표출하는 전역 UI 트리거 로직 추가.
    - 정답 발생 시 영상 정지 및 정답 화면 전환 간의 딜레이를 유기적으로 조정.

### D. 게임 종료 후 '연속 플레이' 흐름 개선
- **현상**: 결과 화면에서 로비로 돌아가면 방이 폭파되거나 다시 방 목록을 거쳐야 함.
- **개선**:
    - 결과 화면에 '대기실로 돌아가기' 버튼 추가.
    - 클릭 시 퇴장이 아닌 `status`를 `WAITING`으로 변경하고 방 정보를 유지하여 바로 다음 판 진행 가능하도록 구현.

### E. 네트워크 탄력성 (Resilience) 확보
- **현상**: 일시적인 네트워크 불안정으로 웹소켓이 끊기면 수동 새로고침이 필요함.
- **개선**:
    - STOMP 클라이언트에 `reconnectDelay`를 설정하고, 최대 재시도 횟수 초과 시 사용자에게 '재연결 시도 중' 안내 메시지 노출.
    - 재연결 성공 시 현재 `roomId`와 `status`를 기반으로 서버 상태를 즉시 재조회(Sync)하는 로직 구현.
