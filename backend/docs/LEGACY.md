# 레거시 코드 및 개선 필요 사항 (LEGACY.md)

본 문서는 현재 `FunGame` 백엔드 프로젝트에서 일관성이 부족하거나, 설계 원칙에 어긋나 개선이 필요한 '레거시' 요소를 정의합니다. 향후 리팩토링 및 신규 기능 개발 시 본 문서를 참고하여 점진적으로 코드를 개선해야 합니다.

---

### 1. 응답 구조의 불일치 (Inconsistent Response Structure)

- **현상**: REST API(`GameController`)는 `ApiResponse` 규격을 엄격히 준수하지만, WebSocket 알림(`GameNotifyService`)은 `Map.of`를 활용해 임의의 필드 구조로 데이터를 전송하고 있습니다.
- **문제점**: 프론트엔드에서 데이터를 처리할 때 HTTP 응답과 WebSocket 메시지의 파싱 로직을 완전히 다르게 작성해야 하며, 공통 에러 처리나 결과 상태 확인이 어렵습니다.
- **개선 방향**: WebSocket 메시지 또한 `ApiResponse` 규격과 유사한 공통 래퍼 클래스를 정의하여 전송해야 합니다.

### 2. 빈약한 도메인 모델 (Anemic Domain Model)

- **현상**: 비즈니스 로직의 상당 부분이 `domain` 패키지의 엔티티가 아닌 `service` 레이어에 집중되어 있습니다. 도메인 객체는 단순히 데이터를 담는 구조체(Getter/Setter 위주)로 사용되는 경향이 있습니다.
- **문제점**: 서비스 클래스가 비대해지고(Fat Service), 도메인 지식이 여러 서비스에 흩어져 캡슐화가 깨지며 유지보수가 어려워집니다.
- **개선 방향**: 비즈니스 규칙과 상태 변경 로직을 도메인 객체(`Game`, `GameRoom` 등) 내부로 이동시켜 '풍부한 도메인 모델(Rich Domain Model)'을 구축해야 합니다.

### 3. 예외 처리 전략의 파편화 (Fragmented Exception Handling)

- **현상**: 전역 예외 처리기(`ApiControllerAdvice`)와 `CoreException` 체계는 마련되어 있으나, 일부 로직에서 표준화되지 않은 일반 예외를 사용하거나 에러 상황에 대한 로그 기록 방식이 일관되지 않습니다.
- **문제점**: 클라이언트에게 정확한 실패 원인을 전달하기 어렵고, 운영 환경에서 문제 발생 시 추적이 힘듭니다.
- **개선 방향**: 모든 비즈니스 예외는 반드시 `ErrorType`을 정의하여 `CoreException`으로 통합하고, 예외 발생 시의 로깅 레벨과 메시지 형식을 표준화해야 합니다.

### 4. 단일 서버 환경의 동시성 제어 전략 부재

- **현상**: 실시간 게임의 특성상 멀티스레드 환경에서의 데이터 정합성이 중요하지만, 현재 `ConcurrentHashMap`을 이용한 메모리 기반 관리 외에 명확한 동기화 전략이 수립되어 있지 않습니다.
- **문제점**: 분산 환경이 아닌 단일 서버 환경이더라도, 여러 사용자가 동시에 동일한 방에 접근할 때 레이스 컨디션(Race Condition)이 발생할 위험이 여전히 존재합니다. (예: 방의 최대 인원을 초과하여 동시에 입장하거나, 동시에 게임 시작 명령을 보내는 경우)
- **개선 방향**: Redis와 같은 분산 락을 사용하는 오버엔지니어링을 지양하고, 자바 표준 라이브러리(`ReentrantLock`, `synchronized` 블록 등) 또는 `Concurrent` 유틸리티를 활용하여 단일 서버 내에서 효율적이고 안전한 동기화 표준을 수립해야 합니다. 특히 도메인 객체 내부에서의 원자적 상태 변경에 집중해야 합니다.

### 5. 테스트 코드의 깊이 부족 (Lack of Test Depth)

- **현상**: Happy Path(성공 케이스) 위주의 테스트는 작성되어 있으나, 예외 상황(Edge Case), 동시성 테스트, 그리고 도메인 로직의 복잡한 비즈니스 규칙에 대한 촘촘한 검증이 부족합니다.
- **문제점**: 리팩토링 시 사이드 이펙트를 완벽히 차단하기 어렵고, 코드의 안정성을 보장하기 힘듭니다.
- **개선 방향**: TDD 원칙을 강화하고, 특히 도메인 레이어의 단위 테스트 커버리지를 극대화해야 합니다.
