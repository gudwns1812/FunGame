# 프론트엔드 개선을 위한 서버 요구사항 (SERVER_REQUIREMENTS.md)

본 문서는 `frontend/guideline.md`에 명시된 기능 개선 및 UX 고도화를 구현하기 위해 백엔드에서 지원해야 할 API 및 기능적 요구사항을 정리합니다.

---

## 1. 플레이어 '준비(Ready)' 상태 관리 시스템

프론트엔드의 공정한 게임 시작을 위해 서버에서 플레이어별 준비 상태를 추적해야 합니다.

- **데이터 모델 변경**: `GamePlayer` 또는 이에 준하는 객체에 `isReady` (boolean) 필드 추가.
- **신규 API**: 
    - `POST /game/rooms/{roomId}/ready`: 현재 플레이어의 준비 상태를 토글(Toggle).
- **WebSocket 이벤트**:
    - `PLAYER_READY_CHANGE`: 누군가 준비 상태를 변경했을 때 방 전체에 알림.
- **비즈니스 로직 수정**:
    - `POST /game/rooms/{roomId}/start` 호출 시, 방장을 제외한 모든 인원이 `isReady == true` 인지 아닌지 웹소켓을 통해 메세지로 전파.

## 2. 실시간 동기화를 위한 '상태 재조회' 기능

네트워크 재연결이나 중도 입장 시 클라이언트가 현재 게임의 정확한 시점을 알 수 있어야 합니다.

- **신규 API (또는 응답 보완)**:
    - `GET /game/rooms/{roomId}/sync`: 현재 진행 중인 라운드 번호, 총 라운드 수, **현재 퀴즈의 서버측 경과 시간**, 비디오 ID 등을 반환.
- **정밀한 타이머 관리**:
    - 서버에서 라운드 시작 시각을 기록하여, 클라이언트 요청 시 정확한 `currentOffset`을 계산해 줄 수 있어야 함 (유튜브 `seekTo` 연동용).

## 3. 정답 피드백 데이터 강화

단순 채팅 외에 시각적 효과를 위해 더 풍부한 데이터를 전송해야 합니다.

- **WebSocket 이벤트 보완 (`CORRECT_ANSWER`)**:
    - 단순히 `playerName`과 `score`만 보내는 것이 아니라, 정답 문구의 정규화된 텍스트(`normalizedAnswer`) 등을 포함하여 프론트엔드에서 애니메이션 처리를 용이하게 함.

## 4. 게임 종료 후 세션 유지 및 복구 (Rematch)

게임 결과 화면에서 대기실로 매끄럽게 돌아가기 위한 로직입니다.

- **비즈니스 로직 수정**:
    - 기존에는 게임이 종료되면 `GameSession`을 삭제하고 방 상태를 `WAITING`으로 바꿨으나, 이때 모든 플레이어의 `isReady` 상태를 `false`로 일괄 초기화해야 함.
- **신규 이벤트**:
    - `BACK_TO_WAITING`: 게임 종료 후 방장이 '대기실로 돌아가기'를 눌렀을 때 발생하는 이벤트.

## 5. 비정상 종료 및 재접속(Re-entry) 처리 고도화

- **세션 타임아웃 유예**:
    - 웹소켓 연결이 끊기자마자 플레이어를 퇴장시키는 것이 아니라, 일정 시간(예: 30초) 동안 '일시 정지' 상태로 유지하여 재접속 시 기존 점수와 세션을 유지할 수 있도록 함.
- **중복 닉네임 정책**:
    - 동일 세션 ID나 토큰을 가진 유저가 다시 들어올 경우, 기존 플레이어 정보를 덮어씌워 '이어하기'가 가능하도록 처리.

---
*본 요구사항은 프론트엔드 구현 우선순위에 따라 순차적으로 백엔드 리팩토링에 반영되어야 합니다.*
