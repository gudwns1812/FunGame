# 작업 계획: 백엔드 리팩토링 보완 및 준비 시스템 완성

## 배경 및 목표
최근 구현된 '플레이어 준비 시스템'의 비즈니스 로직을 강화하고, 도메인 모델과 서비스 레이어 간의 결합도를 낮추기 위해 파라미터 구조를 개선합니다.

## 작업 상세

### 1. 게임 시작 조건 검증 보완 (`GameRoom.java`)
- **작업**: `validateStart` 메서드에 `isAllReady()` 호출을 추가하여 모든 플레이어가 준비되지 않았을 경우 게임 시작을 차단합니다.
- **예외**: 모든 인원이 준비되지 않았을 때 던질 신규 에러 타입(`GAME_ROOM_NOT_ALL_READY`) 정의 및 적용.

### 2. 방장 준비 상태 관리 로직 개선 (`GamePlayers.java`)
- **방 생성 시**: 방장의 `isReady` 상태를 초기값 `true`로 설정.
- **방장 위임 시**: 새로운 방장이 선출될 때 해당 플레이어의 `isReady` 상태를 즉시 `true`로 변경하여 별도의 ready 호출 없이 바로 게임 시작이 가능하게 함.
- **`isAllReady` 로직**: 방장을 포함한 모든 플레이어가 `isReady == true`인 상태를 확인 (방장은 항상 true이므로 실질적으로는 일반 플레이어 체크).

### 3. 서비스 레이어 파라미터 최적화 (`GameRoomService.java` & `GameController.java`)
- **현상**: 컨트롤러에서 서비스로 너무 많은 개별 파라미터를 전달 중.
- **작업**: `createRoom` 메서드의 파라미터를 `CreateRoomRequest` 객체 하나로 통합하거나, 도메인 생성에 필요한 속성들을 응집력 있게 전달하도록 리팩토링.

### 4. 에러 코드 및 응답 동기화 (`ErrorType.java`)
- **신규 에러 추가**: `G009 - 모든 플레이어가 준비되지 않았습니다.` 추가.

## 예상 결과
- 준비되지 않은 플레이어가 있는 상태에서 게임이 시작되는 부정 행위 방지.
- 방장이 바뀌더라도 즉시 게임을 시작할 수 있는 매끄러운 UX 제공.
- 코드의 가독성과 유지보수성 향상.
